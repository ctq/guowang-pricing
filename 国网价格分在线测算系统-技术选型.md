# 国网价格分在线测算系统技术选型

## 1. 选型结论

推荐采用：

| 模块 | 推荐技术 | 结论 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript + Element Plus | 适合表单、表格、参数配置、Excel 上传下载这类后台工具型界面。 |
| 后端 | Python 3.12 + FastAPI | 计算逻辑和 Excel 处理天然适合 Python，FastAPI 适合做类型明确的 API。 |
| 计算引擎 | Python 独立包 + `decimal.Decimal` | 价格分要对账 Excel，必须避免普通浮点数误差。 |
| Excel 导入导出 | openpyxl | 支持 `.xlsx` 读写、单元格样式和多 Sheet 导出。 |
| 数据库 | SQLite 起步，SQLAlchemy 2 ORM | MVP 低成本；后续可平滑迁移 PostgreSQL/MySQL。 |
| 数据校验 | Pydantic v2 | 请求、参数、导入数据统一校验。 |
| 测试 | pytest | 算法对账、边界人数、导出结果都适合自动化测试。 |
| 部署 | Docker Compose + Nginx + Uvicorn/Gunicorn | 单机低成本部署，后续可迁移云服务。 |

一句话判断：这是一个“重计算准确性 + 重 Excel + 中等复杂表单”的工具，不建议做成纯前端，也不建议一开始上复杂微服务。

## 2. 总体架构

```text
浏览器
  |
  | 静态页面 / API 请求 / 文件上传下载
  v
Nginx
  |
  | /api/*
  v
FastAPI 后端
  |
  | 调用
  v
计算引擎 Python 包
  |
  | 读写
  v
SQLite / PostgreSQL

FastAPI 后端同时负责 Excel 导入导出。
```

核心原则：

1. 计算引擎不依赖 Web 框架，后续可以被脚本、后台任务、测试用例复用。
2. 前端只负责交互和展示，不承担最终计算口径。
3. 所有计算结果以后端为准，前端最多做输入预校验。
4. Excel 导入导出放在后端，便于统一字段映射、模板格式和日志追踪。

## 3. 前端选型

### 3.1 推荐：Vue 3 + Vite + TypeScript + Element Plus

适用原因：

1. 系统主要是表单、动态参数、报价表格、结果表格、上传下载，Vue + Element Plus 的开发效率高。
2. Element Plus 的表单、表格、上传、下拉、数字输入、提示组件比较贴合业务。
3. TypeScript 能减少字段名、参数名、接口结构错用。
4. Vite 开发启动快，适合小团队快速迭代。

页面建议：

| 页面/区域 | 说明 |
| --- | --- |
| 测算页 | 单页完成项目信息、算法选择、报价录入、计算结果。 |
| 参数面板 | 根据算法动态展示参数，并提供默认值恢复。 |
| 报价表格 | 支持粘贴 Excel 行列、增删行、标记 `不开标`。 |
| 结果表格 | 展示得分、排名、是否用于基准价、剔除原因。 |
| 计算过程抽屉 | 展示 `A1/A2/区间上下限/回退规则`，便于用户复核。 |

不建议 MVP 使用：

1. 纯 HTML + 原生 JS：初期简单，但动态表格、校验、上传、状态管理很快会变乱。
2. React + Ant Design：也可行，但对这个业务没有明显优势；如果团队更熟 React，可作为替代。
3. Next.js/Nuxt SSR：本系统不是 SEO 内容站，SSR 增加复杂度。

## 4. 后端选型

### 4.1 推荐：FastAPI

适用原因：

1. Python 对 Excel、算法、测试生态更友好。
2. FastAPI 使用类型声明和 Pydantic 校验，请求参数、响应结构、接口文档都更清晰。
3. 性能对本项目足够。单次最多 500 家投标人，计算量很小，瓶颈更多在 Excel 文件解析和网络上传。
4. 后续做批量测算、历史查询、后台任务也容易扩展。

不建议 MVP 使用：

1. Flask：能做，但接口类型、文档和校验需要额外补很多规范。
2. Django：自带能力强，但对当前 MVP 偏重，后台管理和 ORM 不是核心矛盾。
3. Node.js/NestJS：做 Web 很好，但 Excel 对账和十进制计算没有 Python 顺手。

## 5. 计算引擎选型

### 5.1 独立 Python 包

建议目录：

```text
backend/
  app/
    api/
    schemas/
    services/
    db/
  pricing_engine/
    methods/
      a01_interval_average_float.py
      a02_arithmetic_average_float.py
      a03_composite_average_float.py
      a04_second_low_average.py
      a05_lowest_price.py
      a06_simple_average.py
      a07_double_curve.py
      a08_composite_once_average.py
    models.py
    registry.py
    rounding.py
    ranking.py
  tests/
```

计算引擎输入输出：

1. 输入：算法编号、参数、报价列表、最高限价、价格分占比、目标公司。
2. 输出：基准价、折扣率、每家得分、排名、剔除原因、中间计算值。
3. 输出必须包含 `debug`/`trace` 数据，用于导出“计算过程”和定位争议。

### 5.2 数字类型：Decimal

价格分需要与 Excel 对账，不建议用 Python `float` 贯穿全链路。

推荐规则：

1. 所有输入金额转为 `Decimal`。
2. 平均值、基准价、得分计算使用 `Decimal`。
3. 输出时再按规则四舍五入到指定小数位。
4. JSON 响应中金额和得分建议返回字符串或同时返回数值和格式化文本，避免前端二次浮点误差。

## 6. Excel 选型

### 6.1 推荐：openpyxl

适用原因：

1. 支持读取和写入 `.xlsx/.xlsm/.xltx/.xltm`。
2. 能创建多工作表导出文件。
3. 能设置基本样式、列宽、冻结窗格、高亮目标公司和第一名。
4. 对 500 行级别数据性能足够。

注意事项：

1. openpyxl 不能计算 Excel 公式，系统必须自己计算结果后写入值。
2. 不要依赖保存原 Excel 的复杂样式、图片、图表。
3. 导入应只读取数据区域，不执行或信任公式。

### 6.2 为什么不推荐 pandas 作为核心

pandas 适合数据分析，但本项目算法有大量“按人数剔除、区间筛选、并列边界、回退规则、逐行备注”的业务逻辑。直接用 Python 列表和 Decimal 更透明，更容易写对账测试。

pandas 可以作为后续批量统计分析工具，但不作为 MVP 计算核心。

## 7. 数据库选型

### 7.1 MVP：SQLite + SQLAlchemy 2

适用原因：

1. MVP 只沉淀匿名摘要，数据量小。
2. 单机部署最省事，不需要单独数据库服务。
3. SQLAlchemy 能把模型和迁移路径留好，后续切 PostgreSQL/MySQL 成本低。

建议数据表：

| 表 | 用途 |
| --- | --- |
| `calculation_records` | 匿名沉淀摘要。 |
| `calculation_logs` | 计算请求摘要、错误日志、参数快照。 |
| `import_logs` | Excel 导入日志。 |
| `method_defaults` | 算法默认参数，便于后台调整。 |

### 7.2 后续：PostgreSQL

当出现以下情况时迁移 PostgreSQL：

1. 需要多人同时大量使用。
2. 需要历史折扣率查询和统计。
3. 需要用户体系和权限。
4. 需要更强的数据备份、索引、聚合查询。

不建议优先上 MySQL 的原因：PostgreSQL 在 JSON 字段、统计查询、后续分析扩展上更舒服。若团队已有 MySQL 运维经验，也可以用 MySQL。

## 8. 部署选型

### 8.1 MVP 推荐部署

```text
1 台轻量云服务器
  Nginx
  frontend 静态文件
  backend FastAPI 容器
  SQLite 数据文件挂载卷
  定时备份脚本
```

配置建议：

| 配置 | 建议 |
| --- | --- |
| CPU | 2 核 |
| 内存 | 2GB 起步，建议 4GB 更稳 |
| 磁盘 | 40GB SSD |
| 系统 | Ubuntu LTS |
| HTTPS | Nginx + Let's Encrypt |
| 进程 | Docker Compose |

### 8.2 备份

MVP 至少做：

1. SQLite 文件每日备份。
2. 上传文件不长期保存，导入后只保存解析结果摘要。
3. 日志保留 30 到 90 天。

## 9. 测试选型

### 9.1 后端测试

使用 pytest。

必须覆盖：

1. 每种算法的 Excel 模板样例。
2. `M=5/10/20/30` 人数边界。
3. 并列最高价、并列最低价。
4. `不开标`、空报价、非法报价。
5. 区间内无有效报价时的回退规则。
6. 得分负数归零。
7. 排名同分规则。
8. 导出 Excel 工作表和关键字段。

### 9.2 前端测试

MVP 可轻量：

1. 表单参数切换人工测试。
2. 上传、计算、导出主流程人工测试。
3. 后续稳定后再补 Playwright 端到端测试。

不建议 MVP 一开始投入过重的前端自动化，因为主要风险在算法口径和 Excel 对账。

## 10. 安全与隐私

必须做：

1. 上传文件限制大小，例如 10MB。
2. 只允许 `.xlsx`，并校验真实文件类型。
3. 后端不执行 Excel 公式和宏。
4. 默认不保存投标人名称和单家报价明细。
5. 计算日志脱敏，不记录完整报价表。
6. API 做基础限流，防止恶意上传和高频计算。

后续做用户体系时再补：

1. 登录鉴权。
2. 用户级历史记录。
3. 数据删除和导出。
4. 管理后台权限。

## 12. 关键技术风险

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| Excel 模板公式与规则文字不一致 | 计算验收争议 | 每个算法建立固定验收样例，以样例结果为准。 |
| 浮点误差 | 分数无法对账 | 全链路使用 Decimal，输出统一四舍五入。 |
| 并列最高/最低剔除口径不清 | 边界样例失败 | 开发前确认并写入测试用例。 |
| 用户上传非标准开标记录 | 导入失败率高 | MVP 支持标准模板和手动列映射，不承诺任意格式识别。 |
| 数据隐私争议 | 用户不敢使用 | 默认只保存匿名摘要，不保存投标人和报价明细。 |
| 后续功能膨胀 | MVP 延期 | 会员、历史查询、批量多包放到阶段 3。 |

## 13. 最小开发里程碑

### 13.1 第 1 周

1. 搭建 FastAPI 后端、Vue 前端、Docker Compose。
2. 完成计算引擎数据模型、排名、Decimal 工具。
3. 实现 A02、A05 两个最简单算法。
4. 完成手工录入、计算结果展示。

### 13.2 第 2 周

1. 实现 A01、A04。
2. 完成 Excel 导出。
3. 完成匿名沉淀。
4. 补充算法对账测试。

### 13.3 第 3 周

1. 补齐 A03、A06、A07、A08。
2. 完成 Excel 导入和列映射。
3. 补齐边界测试。
4. 部署到测试环境。

## 14. 最终建议

本项目不要追求技术复杂度，核心竞争力在“算得准、解释得清、导出能复核、用户敢上传”。

最终推荐组合：

```text
Vue 3 + Vite + TypeScript + Element Plus
FastAPI + Pydantic + 独立 Python 计算引擎
Decimal + pytest 对账测试
openpyxl 导入导出
SQLite + SQLAlchemy 2
Docker Compose + Nginx 单机部署
```

当日活、数据量和查询需求起来后，再迁移到：

```text
PostgreSQL
Redis 限流/缓存
对象存储临时文件
后台任务队列
用户体系和管理后台
```
